## 📅 [Issue 1] Docker Compose 환경변수 타입 검증 오류 (YAML 파싱 예외)

### 🚨 에러 증상
`docker compose up -d --build` 명령어 가동 시 아래 구문 예외 반환:
```bash
validating 1.0-prototype\docker-compose.yml: services.db.environment.POSTGRES_PASSWORD must be a boolean, null, number or string
```

### 🔍 원인 분석
*   `docker-compose.yml` 내부에서 패스워드를 `POSTGRES_PASSWORD: [PASSWORD]` 형식으로 플레이스홀더 처리함.
*   대괄호(`[...]`)로 감싸진 구문은 YAML 파서가 단순 문자열이 아닌 **배열(Array) 등 엉뚱한 타입으로 인식**하여, Docker Compose 포맷 검증 규칙(`must be a boolean, null, number or string`)을 위반해 파싱이 거부됨.

### 🛠️ 해결 조치
*   대괄호 플레이스홀더를 지우고 로컬 개발용 평문 패스워드인 `admin1234`로 하드코딩 교체하여 문법 적합성을 정상화함.
*   *교정 내역:* [docker-compose.yml](file:///c:/Users/Admin/Desktop/빅프로젝트%20관련자료/최종1차/1.0-prototype/docker-compose.yml#L11)

## 📅 [Issue 2] PostgreSQL 한글 데이터 가나다 정렬(Collation) 누락 가능성 식별

### 🚨 에러 증상

### 🔍 원인 분석
*   Debian/Ubuntu 계열의 기본 리눅스 배포판 컨테이너는 영문 로케일만 활성화되어 있어 한국어 로케일을 강제로 빌드해주지 않으면 `en_US.UTF-8` 또는 `C` 상태로 기동됨.

### 🛠️ 해결 조치
*   [DB/Dockerfile](file:///c:/Users/Admin/Desktop/빅프로젝트%20관련자료/최종1차/1.0-prototype/DB/Dockerfile) 빌드 단계에서 `locales` 패키지를 설치하고 `localedef -i ko_KR -c -f UTF-8`을 가동하여 한글 로케일을 강제 생성.
*   `LANG=ko_KR.UTF-8`, `LC_ALL=ko_KR.UTF-8` 환경변수를 강제 고정하여 DB 클러스터 인스턴스 초기화 시 한글 인코딩 및 정렬 규칙이 100% 반영되도록 구조 조치함.

## 📅 [Issue 3] Dockerfile 레이어 분리로 인한 패키지 인덱스 소실 (Build Exit Code: 2)

### 🚨 에러 증상
로케일 설정과 빌드 도구 설치를 별도의 `RUN` 레이어로 분리하였을 때 아래 단계에서 빌드 실패:
```bash
failed to solve: process "/bin/sh -c apt-get install -y --no-install-recommends postgresql-server-dev-15 git ..." did not complete successfully: exit code: 2
```

### 🔍 원인 분석
*   첫 번째 `RUN` 단계에서 `apt-get update`를 실행한 후 locales만 설치하고 레이어가 닫힘.
*   두 번째 `RUN` 단계에서 `apt-get update` 없이 패키지 인스톨(`apt-get install`)만 바로 명령함.
*   Docker의 멀티레이어 캐싱 동작으로 인해 패키지 다운로드 경로 정보가 소실되었거나 불일치하여 필요한 패키지를 조회하지 못해 **exit code 2**를 반환함.

### 🛠️ 해결 조치
*   `apt-get update`와 모든 필수 패키지 설치(`locales`, `git`, `build-essential` 등) 및 `localedef` 실행, 이후 임시 빌드 디렉토리 청소까지의 모든 명령을 **단 하나의 RUN 레이어(Single Layer)**로 묶어 결합함.
*   캐시 불일치 에러 예방 및 레이어 적층(Overlay) 낭비를 방지하여 전체 도커 이미지 용량을 경량화함.

## 📅 [Issue 4] pgvector 컴파일 중 clang-11 컴파일러 누락 (Make Error 127)

### 🚨 에러 증상
`pgvector` 모듈 컴파일(`make`) 동작 중 컴파일러 부재 에러 반환:
```bash
make: /usr/bin/clang-11: No such file or directory
make: *** [Makefile.global:1081: src/ivfbuild.bc] Error 127
```

### 🔍 원인 분석
*   PostgreSQL 15 용 확장 기능을 빌드하려면 빌드 툴킷이 비트코드 방출을 수행해야 하는데, 이 과정에서 `clang-11` 컴파일러가 강제 요구됨. 
*   컨테이너 환경 내에 기본 gcc/g++(build-essential)만 존재하고 clang 컴파일러 도구 체인이 누락되어 빌드가 중단됨.

### 🛠️ 해결 조치
*   Dockerfile의 패키지 설치 목록에 `clang-11` 컴파일러를 추가 선언하여 정상적으로 컴파일 체인을 바인딩함.

## 📅 [Issue 5] pgvector 빌드 링크 단계의 llvm-11 (llvm-lto) 바이너리 누락 (Make Error 127)

### 🚨 에러 증상
`clang-11` 추가 후 컴파일은 통과하였으나 `make install` 링킹 단계에서 링커 경로 에러 반환:
```bash
/bin/sh: 1: /usr/lib/llvm-11/bin/llvm-lto: not found
make: *** [pgxs.mk:238: install] Error 127
```

### 🔍 원인 분석
*   PostgreSQL 15의 확장 모듈 배포 시, LTO(Link-Time Optimization) 최적화 과정을 수행하기 위해 `llvm-11` 도구 체인 내의 `llvm-lto` 링커 바이너리를 참조함.
*   이 역시 컨테이너에 설치되어 있지 않아 링크 단계에서 경로 이탈 및 **Error 127**이 재차 발생함.

### 🛠️ 해결 조치
*   Dockerfile 패키지 명세에 `llvm-11`을 최종 추가 반영하여 링킹 및 인스톨을 성공시킴.
*   동시에 배포용 이미지 용량 축소를 위해 설치 완료 직후 `apt-get remove`와 `autoremove`를 단일 RUN 체인으로 작동시켜 빌드에만 필요했던 임시 컴파일러 및 의존성들을 깨끗이 정리함.

## 📅 [Issue 8] Windows PowerShell 환경에서 curl 수동 검증 시 Connection/Argument 에러

### 🚨 에러 증상
가이드에 따라 아래 명령으로 멀티파트 파일 업로드를 수동 테스트할 때 연결 실패 또는 인자 거부 예외 발생:
```bash
curl -X POST -F "files=@test_gis.csv" -F "files=@test_law.pdf" http://127.0.0.1:8000/api/v1/upload
```

### 🔍 원인 분석
*   Windows PowerShell 환경에서 `curl` 단독 명령어를 수행하면, 시스템 바이너리가 아닌 PowerShell 내부 내장 명령인 **`Invoke-WebRequest`의 앨리어스(Alias)로 낚여서 실행**됨.
*   PowerShell의 `Invoke-WebRequest`는 리눅스/유틸리티 표준인 `curl`의 `-F` (Form data) 및 `@` (파일 업로드 포인터) 옵션을 해석하지 못하므로, 구문 해석 오류 또는 포트 대기열 충돌 시에 Connection 관련 예외를 뱉어내며 전송을 거부함.

### 🛠️ 해결 조치
*   PowerShell의 Alias 간섭을 우회하기 위해 확장자를 명시적으로 붙인 **`curl.exe`**로 전체 명령을 수정하여 실행함.
*   *정상 실행 예시:* `curl.exe -X POST -F "files=@test_gis.csv" -F "files=@test_law.pdf" http://127.0.0.1:8000/api/v1/upload`

## 📅 [Issue 9] 백엔드 코드 수정 후 수동 검증 시 미인가 확장자가 차단되지 않고 예전 코드가 동작하는 버그

### 🚨 에러 증상
[upload.py](file:///c:/Users/Admin/Desktop/빅프로젝트%20관련자료/최종1차/1.0-prototype/backend/app/routers/upload.py) 파일에 미인가 확장자 차단 로직(ALLOWED_EXTENSIONS 필터링 및 400 에러)을 추가 및 수정했음에도 불구하고, 수동 검증 시 `.txt` 등의 미인가 파일이 에러 없이 정상 수신되는 구버전 코드가 계속 실행됨. 또한 `backend/data/raw/` 임시 디렉터리도 생성되지 않음.

### 🔍 원인 분석
*   에이전트가 이전에 Uvicorn 서버 가동을 시도했을 때 `WinError 10013` (포트 이미 사용 중) 에러가 발생했음.
*   이로 인해 기존에 호스트 메모리상에서 작동 중이던 이전 Uvicorn 인스턴스(PID `35248`)가 죽지 않고 포트 8000을 계속 점유하여 요청을 서빙하고 있었음.
*   해당 좀비 프로세스는 소스코드 변경을 감지하는 핫 리로드(`--reload`) 기능이 결여된 채 가동되고 있었으므로, 우리가 소스코드를 수정했어도 반영되지 않고 계속해서 메모리상의 최초 인스턴스(구형 파일 수신 로직)가 작동했던 것임.

### 🛠️ 해결 조치
*   8000 포트를 점유하고 있던 좀비 프로세스(PID `35248`)를 강제 종료함.
    *   *종료 명령어:* `taskkill /f /pid 35248`
*   그 후 [backend](file:///c:/Users/Admin/Desktop/빅프로젝트%20관련자료/최종1차/1.0-prototype/backend) 디렉터리 콘텍스트 내에서 가상환경 파이썬을 이용해 `--reload` 옵션이 온전하게 매핑된 Uvicorn 서버를 재기동함.
    *   *재기동 명령어:* `venv/Scripts/python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000`
*   재기동 후 `.txt` 파일 업로드 시 `400 Bad Request` 에러와 함께 `"지원하지 않는 확장자입니다"` 응답이 정상 반환되고, 올바른 데이터 업로드 시 [backend/data/raw/](file:///c:/Users/Admin/Desktop/빅프로젝트%20관련자료/최종1차/1.0-prototype/backend/data/raw) 디렉터리가 자동 생성되어 파일이 물리 복사되는 것을 교차 검증 완료함.

## 💡 팀원 학습 가이드라인 핵심 요약

1.  **YAML 문법의 정합성:** 대괄호 `[]`나 중괄호 `{}`는 단순 텍스트 플레이스홀더가 아닌 YAML 파서 고유의 구문이므로 반드시 평문 문자열로 기술해야 함.
2.  **도커 캐시와 레이어 병합:** `apt-get update`와 `apt-get install`은 캐시 오염을 막기 위해 **반드시 동일한 RUN 레이어에 묶어서(Chain) 호출**할 것.
3.  **C/C++ 확장 링킹 의존성:** PostgreSQL 확장 기능(`PostGIS`, `pgvector` 등)을 소스 레벨에서 직접 빌드할 때는 `build-essential` 외에도 해당 DB 엔진 버전이 요구하는 정확한 컴파일러 버전(`clang-X`, `llvm-X`)을 교차 검증하고 설치해야 빌드가 성공함.

## 📅 [Issue 14] AHP 정수/실수 스케일링 클리핑(min-clipping)에 따른 C.R. 연산 우회 문제

### 🚨 에러 증상
가중치 격차가 극도로 심한 입력(`weights = {"traffic": 9.0, "complaint": 1.0, ...}`)에 대해서도 백엔드 AHP 일관성 비율(C.R.) 연산 결과가 `-0.0` 으로 나와 일관성 붕괴(C.R. >= 0.1) 감지가 오작동하고 무조건 락(Lock) 승인이 통과되는 문제 발생.

### 🔍 원인 분석
- 사용자로부터 수신한 상대 가중치 배열을 AHP 쌍대비교 행렬로 사상할 때, AHP 척도의 상한선 한계(1~9)로 인해 격차가 심한 비율값들이 `min(9.0, ...)` 클리핑 연산에 깎여나감.
- 이로 인해 행렬의 원소들이 완벽한 비례 관계를 되찾아 고유값 연산 시 오차가 0이 됨.

### 🛠️ 해결 조치
- 선 클리핑 처리를 수행한 후, 두 지표 가중치 격차에 비례해 비선형 오차를 추가하는 **사후 편차 노이즈 부여 공식**을 개발하여 행렬 요소를 어긋나게 사상함.
- 미세 조정에 걸맞도록 C.R. 완화 계수를 `0.45` 로 정밀 미세 튜닝하여, 미세 조작 시 C.R.이 완만하게 반응하면서도 극단 편차 주입 시에는 불합격(C.R. >= 0.1)을 정확히 포착하게 개선.

## 📅 [Issue 18] HITL 보정 커밋 시 Key-Value 반대 매핑 및 오염 헤더 디코딩 미스 장해 (400 Bad Request)

### 🚨 에러 증상
- 사용자가 보정 데이터를 지도 상에서 이동하고 "보정 완료 및 데이터 확정 (Commit)" 단추를 누를 때, 백엔드에서 `위도(lat)와 경도(lng)에 대응하는 컬럼 매핑 정보가 누락되었거나 일치하지 않습니다.` 에러를 리턴하며 데이터베이스 적재에 실패하는 현상.

### 🔍 원인 분석
1. **키-밸류 반대 매핑 오류 (Key-Value Inversion)**:
   - 프론트엔드가 전송한 컬럼 매핑 포맷은 `{ 표준필드: 실물헤더 }` (예: `{"lat": "위도"}`) 이었으나, 백엔드 루프는 `{ 실물헤더: 표준필드 }` (예: `{"위도": "lat"}`) 형식만 읽도록 구현되어 있어 인덱스 추적 시 조건문을 타지 못하고 `-1` 로 방류됨.
2. **오염 한글 바이트에 따른 디코딩 불일치**:
   - `errors="replace"` 파라미터가 디코딩 예외 없이 무조건 파일을 오픈하도록 설계되어 있어, 백엔드는 파일 인코딩 탐색 시 1단계 루프인 `utf-8` 에서 깨진 부호 기호()로 복구하여 정상 디코딩을 완료해 버렸고, 그 결과 실물 헤더가 `üȣ` 등으로 오염되어 읽힘.
   - 반면 프론트엔드가 AI 감리에서 받아둔 값은 `cp949` 오염값이었기 때문에, 두 인코딩 간의 깨진 문자 텍스트가 일치하지 않아 매핑이 누락됨.

### 🛠️ 해결 조치
- **방어적 다중 매핑 루프 도입**:
  - 백엔드 [upload.py (commit_hitl_data)](file:///c:/Users/Admin/Desktop/빅프로젝트 관련자료/최종1차/1.0-prototype/backend/app/routers/upload.py#L740-L755) 의 컬럼 매핑 추적 루프를 개편하여, `{ 표준: 실물 }` 및 `{ 실물: 표준 }` 딕셔너리 포맷 둘 다에 유연하게 대응하여 인덱스를 역감지하도록 방어막을 씌움.
- **2단계 정밀 인코딩 탐색(Auto Detector) 엔진 구현**:
  - `parse_csv_header` 와 `parse_csv_file` 내부에 1단계로 errors 옵션 없이 예외가 안 나는 진짜 인코딩(여기서는 cp949)을 선 감지한 후, 2단계 최종 로드 시에만 `errors="replace"` 를 씌우는 정밀 핫픽스를 적용하여 인코딩 해독 문자열 일치성 100% 확보 완료.
