# [연구노트] 지능형 다목적 스마트시티 입지 선정 및 공공갈등 예측 플랫폼 'OmniSite' 연구개발노트 (v1.0.0-prototype)

## 1. 개요 및 연구 목적
본 연구노트는 **"지능형 다목적 스마트시티 입지 선정 및 공공갈등 예측 플랫폼 OmniSite"**의 구상 단계부터 개발 착수(v1.0.0-prototype)까지 조장/PM 배종현과 AI 어시스턴트(Antigravity) 간의 기술 아키텍처 토론, 위기극복 과정, 그리고 핵심 설계 변경 이력을 체계적으로 기록한 문서입니다.

## 2. 📅 버전별 연구개발 이력 및 마일스톤

### [Root] 기초 수집 단계
*   **연구 내용:** 용산구 중심의 실외 흡연구역 선정을 PoC로 설정하고 관련 공공데이터 19종(지적도, 버스정류소, 학교, 상가 등)을 수집하여 포맷 통일.
*   **주요 의사결정:** GIS 공간 차집합 필터링과 AHP 다기준 의사결정 모델을 접목한 스마트시티 입지 선정의 타당성 정립.

### [0701] 1차 사업 구상 단계
*   **연구 내용:** 수집된 실물 데이터를 기반으로 입지 선정 알고리즘 초안 설계.
*   **주요 의사결정:** 10m 대중교통 금역 규제 및 200m 학교 정화구역을 차집합으로 배제하는 PostGIS 쿼리 설계.

### [0.1] 정합성 감리 단계
*   **연구 내용:** 데이터의 주관적 정합성을 검증하기 위한 AI 데이터 감리 엔진 및 HITL 설계.
*   **주요 의사결정:** 데이터 누락이나 오차가 행정적 하자로 이어지지 않도록 2단계에서 AI가 사전 감리하고 공무원이 승인(HITL)하는 보안 안전장치 추가.

### [0.2] 시뮬레이션 및 피드백 루프 단계
*   **연구 내용:** 선정된 입지에 대한 주민 반발 리스크를 예측하기 위한 LangGraph 기반 페르소나 모의 토론 모델 도입.
*   **주요 의사결정:** 
    - AI가 스스로 생성한 가상 예측 사례가 다시 RAG로 피드백될 때 발생하는 **모델 붕괴(Model Collapse)** 방지 대책 수립.
    - 실제 결과 등록 시 최종 결재 공문서(PDF) 검증을 대조하는 **Audit AI** 세그먼트 완전 격리 설계.

### [0.3] 고도화 및 인프라 표준화 단계
*   **연구 내용:** B2G 다목적 플랫폼으로서의 명칭 승격 및 Next.js + FastAPI + AWS EKS 인프라 구조 확립.
*   **주요 의사결정:** Multi-Tenancy 구성을 위한 EKS Namespace 격리 및 AWS RDS(PostGIS) 성능 튜닝 표준 수립.

### [0.4] 아키텍처 다각화 및 실무 알고리즘 정립
*   **연구 내용:** 메인 백엔드를 **Java Spring Boot 또는 Pocketbase**로 이원화하고 Python FastAPI를 AI/GIS 마이크로서비스로 배치하는 하이브리드 아키텍처 설계.
*   **주요 의사결정:**
    - **지역 갈등 민감도 벡터 ($CSS(p)$) 산출 수학적 공식**을 공식 지정 및 제어 인자로 연동.
    - **Top 1~3 부지 디렉토리/탭 구조의 UX 인터페이스 설계:** Top 1은 자동 시뮬레이션, Top 2 & 3은 온디맨드 버튼 이벤트 방식으로 동작하게 하여 API 호출 비용을 합리적으로 제어하고 토론 품질(GPT-4o)을 확보.

### [0.5] 백엔드 단일화 통합 및 개발 가속도 확보 (현재 단계)
*   **연구 내용:** 자바 에이스의 리스크(SPOF)를 분산하고 3-Tier 연동의 비효율성을 해소하기 위해 백엔드를 **Python FastAPI 단일 통합형(Monolith)**으로 단일화.
*   **주요 의사결정:**
    - Next.js ➔ FastAPI로의 직접 통신 구축으로 네트워크 대기 지연 최소화.
    - 데이터, AI, 백엔드 개발 스택을 Python으로 통일하여 상호 코드 리뷰 및 백업 가능 구조 확립.
    - 실시간 토론 전문 출력을 위해 FastAPI에서 Next.js 브라우저로 SSE(Server-Sent Events) 스트리밍 다이렉트 전송 구현.

*   **연구 내용:** Step 4 진입 시 발생하는 브라우저 스크립트 중복 로딩 오류 정정 및 맵 조작 차단 설계 고도화.
*   **주요 의사결정:**
    - **Leaflet 로더 비동기 분리 개편:** 기존에 `pipelineStep`이 변경될 때마다 매번 `<script>` 엘리먼트를 동적으로 재생성하여 `window.L` 전역 컨텍스트를 덮어쓰던 문제를 전면 제거. `useEffect` 마운트 시 최초 1회만 스크립트를 로드해 `leafletLoaded` 상태를 공유하는 싱글톤 라이프사이클로 리팩토링하여 **Step 4 진입 시의 메모리 릭 및 크래시 에러를 완벽 조치**.

    - **PostGIS-RAG 하이브리드 설계 문서화:** 앞선 피어 토의에 근거해 위치 연산(PostGIS)과 조례 해석(RAG)의 하이브리드 아키텍처 명세 추가.

### [1.0.0-prototype-Rev4] 지도 인터랙션 복원 및 마커 드래그 락(Lock) 조율 단계
*   **연구 내용:** 지도 완전 고정 조치로 인한 일반 줌/드래그 불가 및 마커 드래그 메커니즘 오작동 버그 교정.
*   **주요 의사결정:**
    - **지도 드래그/줌 복원 및 줌컨트롤 표출:** 지도 초기화 시 `dragging: true`, `scrollWheelZoom: true`를 복구하고 우측 하단 줌 버튼을 재도입하여 공무원이 원하는 크기로 확대/축소하여 필지를 검수할 수 있도록 자유도 보장.
    - **드래그 스크롤 차단(autoPan: false) 및 드래그 중 지도 락:** 마커 객체 속성에 `autoPan: false`를 부여하여 마커를 지도 가장자리로 당겨도 지도가 멋대로 스크롤되는 것을 방지하고, 마커의 `dragstart` 및 `dragend` 주기에 맞춰 `map.dragging.disable()`과 `map.dragging.enable()`을 동적으로 수행하도록 재설계. 이로써 **평소에는 자유롭게 줌/드래그를 하다가 마커를 옮길 때만 지도가 고정되는 무결점 드래그 감도** 구현 완료.

### [1.0.0-prototype-Rev5] 핫 리로드 인스턴스 영속성 해제 및 드래그 스로틀링 튜닝
*   **연구 내용:** 핫 리로드 시 React Ref 잔존에 따른 지도 정지 버그와 드래그 도중 발생하는 DOM 쓰레싱 렉 조치.
*   **주요 의사결정:**
    - **맵 리프레시 갱신 잠금 해제:** 컴포넌트 갱신 시 `mapRef.current`가 메모리에 이미 유지되는 경우 신규 지도 초기화 옵션(`dragging: true`)이 적용되지 않던 문제를 교정하기 위해, `useEffect` 진입 시 기존 맵 인스턴스를 즉각 찾아내 수동으로 드래그 및 줌 속성들을 강제 활성화(`map.dragging.enable()`)하도록 보완.
    - **드래그 쓰레싱 렉 차단 (핵심 성능 개선):** 마커 드래그 시 매 프레임(60fps)마다 `marker.setIcon()`이 연속 호출되어 DOM 트리가 끊임없이 폭발/재생성되면서 브라우저가 드래그 타깃을 잃고 핀 움직임이 끊기던 버그를 식별. 마커 객체 내부 경고 상태(`marker.isWarning`) 플래그를 두어 **안전 구역에서 위험 구역으로 국면이 '전환'되는 단 1회에만 setIcon이 트리거**되도록 성능을 최적화. 이로써 핀 드래그 감도의 초고속 반응성과 렉 제로 반응을 완비 완료.
